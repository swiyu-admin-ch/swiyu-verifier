# Migration Guide: v3.0.x → v4.0.0

This guide helps you migrate from **v3.0.x** to **v4.0.0** of the Swiyu Verifier Service.

It is based on the **v4.0.0** section in `CHANGELOG.md` and includes breaking changes related to
configuration properties, cache management, encryption algorithms, and verification flow behavior.

## Compatibility summary

- **Wallets / OID4VP clients:** 
  - The `client_id` format has changed due to the introduction of a configurable prefix.
  - The `oauthState` parameter is now **mandatory** in verification responses.
- **Business Verifiers:**
  - Empty `values` lists in `DcqlClaimDto` are now rejected (spec compliance).
- **Operators / DevOps:** 
  - Configuration property `client_id_scheme` has been removed and replaced with `client_id_prefix`.
  - Environment variables `SWIYU_TRUST_REGISTRY_CUSTOMER_KEY` and `SWIYU_TRUST_REGISTRY_CUSTOMER_SECRET` 
    are no longer used (read-only trust registry does not require authentication).
  - Cache configuration has been unified: `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI` is deprecated in favor 
    of `JWK_CACHE_TTL_MILLI`.
  - Encryption algorithm upgraded from A128GCM to A256GCM for enhanced security.
  - Container images are now signed with Cosign for supply chain security.

## Breaking changes

Work through every applicable item in this section.

### 1. Client ID scheme configuration: `client_id_scheme` → `client_id_prefix`

**Before (v3.0.x):** The verifier used a hard-coded `client_id_scheme` configuration property 
to construct the `client_id` used in OAuth2/OIDC flows.

**After (v4.0.0):** The `client_id_scheme` property has been **removed** and replaced with 
`client_id_prefix`, which defaults to `decentralized_identifier`. The `client_id` is now 
constructed as: `${client_id_prefix}:${client_id}`.

**Action required:**

1. **If you never explicitly set `client_id_scheme`** in your configuration: no action needed. 
   The new default `client_id_prefix=decentralized_identifier` can be handled by the wallet.

2. **If you customized `client_id_scheme`**: update your configuration files 
   (`application.yml`, `application-prod.yml`, Kubernetes ConfigMaps, Helm values, etc.):

   ```yaml
   # Before (v3.0.x)
   verifier:
     client_id_scheme: "custom_scheme"
   
   # After (v4.0.0)
   verifier:
     client_id_prefix: "custom_scheme"
   ```

3. **If you want to disable the prefix entirely** (resulting in `client_id` without a prefix), 
   set the value to `null` and reproduce the old behavior:

   ```yaml
   verifier:
     client_id_prefix: null
   ```

**Testing:** After upgrading, verify that wallets can still complete the verification flow by 
inspecting the `client_id` sent in authorization requests. It should now follow the pattern 
`<prefix>:<id>` (or just `<id>` if prefix is `null`).

### 2. Trust Registry authentication removed: `SWIYU_TRUST_REGISTRY_CUSTOMER_KEY` / `SWIYU_TRUST_REGISTRY_CUSTOMER_SECRET`

**Before (v3.0.x):** The verifier supported optional authentication to the trust registry using 
environment variables `SWIYU_TRUST_REGISTRY_CUSTOMER_KEY` and `SWIYU_TRUST_REGISTRY_CUSTOMER_SECRET`.

**After (v4.0.0):** The trust registry integration now operates in **read-only mode** and does not 
require authentication. These environment variables are **no longer used** and can be safely removed.

**Action required:**

Remove the following environment variables from your deployment manifests:

```yaml
# Remove from docker-compose.yml, Kubernetes Deployment, Helm values, etc.
# environment:
#   SWIYU_TRUST_REGISTRY_CUSTOMER_KEY: "..."
#   SWIYU_TRUST_REGISTRY_CUSTOMER_SECRET: "..."
```

**Testing:** Confirm that trust registry queries still succeed after removing the credentials. 
Monitor logs for any authentication-related errors (there should be none).

### 3. Cache configuration unified: `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI` → `JWK_CACHE_TTL_MILLI`

**Before (v3.0.x):** The verifier maintained two separate caches (`JWK_CACHE` and `PUBLIC_ISSUER_KEY_CACHE`) 
with separate TTL configuration via `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI`.

**After (v4.0.0):** The caches have been **unified into a single `JWK_CACHE`**. The environment variable 
`ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI` is **deprecated** but still accepted as a fallback for 
`JWK_CACHE_TTL_MILLI` to maintain backward compatibility.

**Action required:**

1. **If you never set `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI`**: no action needed.

2. **If you customized `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI`**: rename it to `JWK_CACHE_TTL_MILLI`:

   ```yaml
   # Before (v3.0.x)
   environment:
     ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI: "3600000"
   
   # After (v4.0.0)
   environment:
     JWK_CACHE_TTL_MILLI: "3600000"
   ```

**Note:** Existing deployments that still use `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI` will continue to 
work in v4.0.0 (the value is used as a fallback), but you should migrate to the new variable name 
as the old one will be removed in a future release.

### 4. Mandatory `oauthState` in verification responses

**Before (v3.0.x):** The `oauthState` parameter in verification responses was optional in some scenarios.

**After (v4.0.0):** The `oauthState` parameter must **always be sent** in verification responses. 
The verifier will **reject responses that omit this parameter**.

**Impact:** This change primarily affects **wallet implementations**. If you operate a wallet or 
test client that interacts with the verifier, ensure it always includes `oauthState` in responses.

**Action required (for wallet developers):**

Ensure your wallet sends the `oauthState` parameter in every verification response

### 5. Encryption algorithm upgraded: A128GCM → A256GCM

**Before (v3.0.x):** The verifier used **A128GCM** (128-bit AES-GCM) for JWE encryption.

**After (v4.0.0):** The verifier now uses **A256GCM** (256-bit AES-GCM) for enhanced security.

**Impact:** 
- **Backward compatibility:** JWE payloads encrypted with A128GCM by v3.0.x **may not be decryptable** 
  by v4.0.0 (and vice versa) if the encryption algorithm is part of the negotiation.
- **Forward compatibility:** Ensure that clients (wallets, business verifiers) that decrypt JWE 
  responses from the verifier support A256GCM.

**Action required:**

1. **Check client capabilities:** Verify that all clients consuming JWE-encrypted responses from the 
   verifier support the A256GCM algorithm.

2. **Rotation strategy (if stateful encryption is used):** If you have long-lived encrypted objects 
   in your database or cache that were created with A128GCM, plan a migration.

### 6. Empty `values` in `DcqlClaimDto` now rejected

**Before (v3.0.x):** Business verifiers could send a `DcqlClaimDto` with an empty `values` list, 
which violated the DCQL specification but was not enforced.

**After (v4.0.0):** The verifier now **validates** that the `values` list is not empty and will 
**reject requests** that violate this constraint.

**Impact:** Business verifiers that accidentally or intentionally sent empty `values` lists will 
now receive validation errors.

## Non-breaking enhancements

The following changes improve security and observability but do **not** require action:

### Container image signing with Cosign

All published container images (both hardened and unhardened variants) are now **cryptographically 
signed** using [Cosign](https://docs.sigstore.dev/) with keyless OIDC signing. Signatures are bound 
to the immutable image digest and published to the Sigstore transparency log.

**Optional action (recommended):** Verify image authenticity before deploying:

```bash
cosign verify \
  --certificate-identity-regexp "https://github.com/swiyu-admin-ch/swiyu-verifier" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  ghcr.io/swiyu-admin-ch/swiyu-verifier:4.0.0
```

### Dependency PGP signature verification

The build now uses `pgpverify-maven-plugin` to cryptographically verify PGP signatures of all 
third-party dependencies. The build fails if an artifact has no signature or an invalid signature.

**Impact:** Developers and CI/CD pipelines may see slightly longer build times due to PGP key 
downloads. Keys are cached in CI/CD to mitigate this.

### Additional verification purpose validation

The verifier now checks that `verification_purpose.purpose_name` and 
`verification_purpose.purpose_description` contain the necessary default keys.

**Impact:** Misconfigured verification purposes will now fail earlier with clearer error messages.

### SD-JWT algorithm validation

The verifier now validates the `_sd_alg` value in SD-JWT tokens to ensure it is a supported 
algorithm (currently only `sha-256` is supported).

**Impact:** SD-JWTs using unsupported hash algorithms will now be rejected explicitly.

### Trust statement caching

TTL-based cache eviction for trust statements has been added (until a proper caching concept is 
implemented ecosystem-wide).

**Impact:** Improved performance when resolving trust statements repeatedly.

### State parameter in `direct_post.jwt`

The `state` parameter provided in the response is now also accepted when using `direct_post.jwt` 
as part of the JWT.

**Impact:** Enhanced interoperability with wallets that embed `state` in the JWT payload.

## Migration checklist

### Configuration changes

- [ ] Search for `client_id_scheme` in your configuration files and rename it to `client_id_prefix`.
- [ ] Remove `SWIYU_TRUST_REGISTRY_CUSTOMER_KEY` and `SWIYU_TRUST_REGISTRY_CUSTOMER_SECRET` from 
      deployment manifests.
- [ ] Rename `ISSUER_PUBLIC_KEY_CACHE_TTL_MILLI` to `JWK_CACHE_TTL_MILLI` (or leave as-is temporarily, 
      but plan migration).

### Client/wallet changes

- [ ] Ensure wallets **always send `oauthState`** in verification responses.
- [ ] Verify that wallets support **A256GCM encryption** and can decrypt JWE responses.
- 
### Business verifier changes

- [ ] Ensure DCQL queries never send an **empty `values` list** in `DcqlClaimDto`.
